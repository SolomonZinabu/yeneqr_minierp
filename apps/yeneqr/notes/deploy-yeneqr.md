### Build issue discovered
`npm run build` (production standalone) is silently broken in the repo.
The `build` script ends with `; true` which swallows the failure.
Root cause: `next-themes@0.4.6` + Next.js 16.1.3 hits
`TypeError: Cannot read properties of null (reading 'useContext')`
during static prerender of `/_global-error` (and previously `/_not-found`).
A `not-found.tsx` with `export const dynamic = 'force-dynamic'` was added
and fixes the `_not-found` page, but `_global-error` still fails (the
`dynamic` export does not apply to client components). To unblock preview,
we run `next dev` via PM2 instead. This build issue should be filed as a
separate bug to fix before the next production deploy.

## Critical PM2 env-var gotcha (DO NOT FORGET)

The repo ships `ecosystem.config.js` at the project root with
`DATABASE_URL=file:/home/z/my-project/db/custom.db` baked in.
PM2 v7 AUTO-LOADS this file when starting any process whose cwd is
the project root, OVERRIDING any env vars you set via `--env KEY=VAL`
or shell exports.

WORKAROUND: rename the repo's ecosystem files before starting PM2:
  mv ecosystem.config.js ecosystem.config.js.orig
  mv ecosystem.config.prod.js ecosystem.config.prod.js.orig

Then start with explicit env:
  DATABASE_URL="file:/home/z/my-project/YeneQR_token2/db/yeneqr.db" \
  JWT_SECRET="..." QR_SECRET="..." \
  pm2 start node_modules/next/dist/bin/next --name yeneqr -- dev -p 3000

(Already done in this preview. Rename back to .js before deploying
to production — production uses ecosystem.config.prod.js which is
generated by deploy.sh.)
